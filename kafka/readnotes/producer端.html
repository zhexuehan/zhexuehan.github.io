Kafka Producer

生产者消息分区

Kafka 的消息组织是三级结构：主题 - 分区 - 消息。主题下的每条消息只会保存在某一个分区中，而不会在多个分区中被保存多份：

分区的作用就是提供负载均衡能力，实现系统的高伸缩性，利用分区也可以实现业务级别的消息顺序的问题。

分区策略

自定义分区，需要实现org.apache.kafka.clients.producer.Partitioner接口，显式地配置生产者端的参数partitioner.class的Full Qualified Name。

随机策略

按消息键保序策略：Kafka 允许为每条消息定义消息键，一旦被定义了 Key，可以保证同一个 Key 的所有消息都进入到相同分区。Kafka 默认分区策略同时实现了两种：如果指定了 Key，按消息键保序策略；否则使用轮询策略。

地理位置分区策略：针对大规模跨地区和国家来说

生产者压缩算法

Kafka有量大数据压缩版本，V1和V2, 0.11.0.0正式引入V2版本，需要特别注意。

V2与V1相比，把消息的公共部分抽取出来放到外层消息集合里面，不用每条消息都保存这些信息。比如V1中的CRC，在Broker端可能会对时间戳改变或者为了兼容之前消息格式进行重新转换，都会导致重新计算，因此V2中，CRC被移交到了消息集合这层。另外V1是对多个消息压缩保存在外层消息体中，V2对整个消息集合压缩，效率更高。

何时压缩：生产者端和 Broker 端。

生产者程序中配置 compression.type 参数即表示启用指定类型的压缩算法，可有效节约网络带宽。例如Java中的

“props.put("compression.type", "gzip");”

Broker端压缩和Procuder端压缩算法不同，或者Broker端为了兼容老版本消息格式，需要重新解压和再次压缩，也会丧失零拷贝特性。

何时解压缩：Consumer端和Broker端。

Consumer端不必说，Broker端除了消息格式转换解压缩，为了执行必要消息体校验，也会对消息结合进行解压缩校验，这是必须的。

Kafka消息无丢失配置

Kafka 只对“已提交”的消息（committed message）做有限度的持久化保证。

  1. Kafka 若干个 Broker 成功接收一条消息并写入日志文件后，告诉生产者程序消息已成功提交。这条消息在 Kafka 看来就正式变为“已提交”消息了。具体几个Broker取决与于定义(ISR)。

  2. Kafka不保证在任何情况下都做到不丢失消息，例如所有Broker不可逆毁坏。

消息保存在 N 个 Kafka Broker 上，这 N 个 Broker 中至少有 1 个存活。Kafka 就能保证这条消息永远不会丢失。

消息丢失的可能情况

生产者丢失消息：生产者异步发送消息。Producer要使用带有回调通知的发送 API，要使用 producer.send(msg, callback)，对发送失败用针对出来。

消费者程序丢失：消费位移的错乱。应该维持先消费消息，再更新位移的顺序即可。另外一种情况是消费端的多线程处理，某个线程没有消费成功，但是位移信息被其他线程更新，如果是多线程异步处理消费消息，Consumer 程序不要开启自动提交位移，而是要应用程序手动提交位移，且是依次有序提交。

最佳实践：

  1. 不要使用 producer.send(msg)，而要使用 producer.send(msg, callback)。记住，一定要使用带有回调通知的 send 方法。
  2. 设置 acks = all。acks 是 Producer 的一个参数，代表了对“已提交”消息的定义。如果设置成 all，则表明所有副本 Broker 都要接收到消息，该消息才算是“已提交”。这是最高等级的“已提交”定义。
  3. 设置 retries 为一个较大的值。这里的 retries 同样是 Producer 的参数，对应前面提到的 Producer 自动重试。当出现网络的瞬时抖动时，消息发送可能会失败，此时配置了 retries > 0 的 Producer 能够自动重试消息发送，避免消息丢失。
  4. 设置 unclean.leader.election.enable = false。这是 Broker 端的参数，它控制的是哪些 Broker 有资格竞选分区的 Leader。如果一个 Broker 落后原先的 Leader 太多，那么它一旦成为新的 Leader，必然会造成消息的丢失。故一般都要将该参数设置成 false，即不允许这种情况的发生。
  5. 设置 replication.factor >= 3。这也是 Broker 端的参数。其实这里想表述的是，最好将消息多保存几份，毕竟目前防止消息丢失的主要机制就是冗余。
  6. 设置 min.insync.replicas > 1。这依然是 Broker 端参数，控制的是消息至少要被写入到多少个副本才算是“已提交”。设置成大于 1 可以提升消息持久性。在实际环境中千万不要使用默认值 1。
  7. 确保 replication.factor > min.insync.replicas。如果两者相等，那么只要有一个副本挂机，整个分区就无法正常工作了。我们不仅要改善消息的持久性，防止数据丢失，还要在不降低可用性的基础上完成。推荐设置成 replication.factor = min.insync.replicas + 1。
  8. 确保消息消费完成再提交。Consumer 端有个参数 enable.auto.commit，最好把它设置成 false，并采用手动提交位移的方式。就像前面说的，这对于单 Consumer 多线程处理的场景而言是至关重要的。

Kafka客户端的拦截器

Kafka 拦截器分为生产者拦截器和消费者拦截器。

实现“org.apache.kafka.clients.producer.ProducerInterceptor”或者“org.apache.kafka.clients.consumer.ConsumerInterceptor”接口，通过Procucer或者Consumer端的interceptor.classes参数指定全限定名。

Producer端TCP链接管理

Producer生命周期：

  1. 构造生产者对象所需的参数对象。

  2. 利用参数对象，创建 KafkaProducer 对象实例。

  3. 使用 KafkaProducer 的 send 方法发送消息。

  4. 调用 KafkaProducer 的 close 方法关闭生产者并释放各种系统资源。

何时创建TCP链接

在创建 KafkaProducer 实例时，生产者会在后台创建启动一个 Sender 线程，该线程开始运行时首先会创建与 Broker的连接。

问题：如果不调用 send 方法，Producer 都不知道给哪个主题发消息，它又怎么知道连接哪个 Broker 呢？

会连接 bootstrap.servers 参数指定的所有 Broker 。bootstrap.servers是指定了这个 Producer 启动时要连接的 Broker 地址。实际使用中，不建议把集群中所有 Broker 信息都配置到 bootstrap.servers 中，通常指定 3～4 台就足以了。因为 Producer 一旦连接到集群中的任一台 Broker，就能拿到整个集群的 Broker 信息。

《Java 并发编程实践》明确指出这样做的风险：在对象构造器中启动线程会造成 this 指针的逃逸。理论上，Sender 线程完全能够观测到一个尚未构造完成的 KafkaProducer 实例。当然，在构造对象时创建线程没有任何问题，但最好是不要同时启动它。

TCP 连接还可能在两个地方被创建：一个是在更新元数据后，另一个是在消息发送时。

当 Producer 更新了集群的元数据信息之后，发现与某些 Broker 当前没有连接，那么它就会创建一个 TCP 连接。

场景一：当 Producer 尝试给一个不存在的主题发送消息时，Broker 会告诉 Producer 说这个主题不存在。此时 Producer 会发送 METADATA 请求给 Kafka 集群，去尝试获取最新的元数据信息。

场景二：Producer 通过 metadata.max.age.ms 参数定期地去更新元数据信息。

合适关闭TCP链接

一种是用户主动关闭：调用Kill -9 或者显示调用producer.close() 方法来关闭。

一种是 Kafka 自动关闭：与Producer 端参数 connections.max.idle.ms 的值有关。默认是 9 分钟，即在 9 分钟内没有任何请求“流过”某个 TCP 连接，那么 Kafka 会主动把该 TCP 连接关闭。用户可以在 Producer 端设置 connections.max.idle.ms=-1 禁掉这种机制。一旦被设置成 -1，TCP 连接将成为永久长连接，但keepalive 探活机制还是会遵守。TCP 连接是在 Broker 端被关闭，但其实这个 TCP 连接的发起方是客户端，因此在 TCP 看来，这属于被动关闭的场景。被动关闭的后果就是会产生大量的 CLOSE_WAIT 连接，因此 Producer 端或 Client 端没有机会显式地观测到此连接已被中断。

幂等生产者和事务生产者

消息交付可靠性包括：最多一次、至少一次、精确一次。Kafka默认使用第二种，但也可以提供最多一次保障，只需要关闭Producer端失败重试即可。

幂等性Producer

Kafka 0.11.0.0引入新功能，可以创建幂等性Producer。指定 Producer 幂等性的方法仅需要设置一个参数即可，即 props.put(“enable.idempotence”, ture)，或 props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG， true)。

底层具体的原理也简单，用空间换时间优化思路，在 Broker 端多保存一些字段。当 Producer 发送了具有相同字段值的消息后，Broker 能够自动知晓这些消息已经重复了，于是可以在后台默默地把它们“丢弃”掉，大致可以这么理解。

幂等性 Producer 的作用范围

首先它只能保证单分区上的幂等性，即一个幂等性 Producer 能够保证某个主题的一个分区上不出现重复消息，它无法实现多个分区的幂等性。其次，它只能实现单会话上的幂等性，不能实现跨会话的幂等性。这里的会话，可以理解为 Producer 进程的一次运行。当重启了 Producer 进程之后，这种幂等性保证就丧失了。

如果想实现多分区以及多会话上的消息无重复，需要使用事务（transaction）或者依赖事务型 Producer。

事务型Producer

Kafka 自 0.11 版本开始提供对事务的支持，目前主要是在 read committed 隔离级别上做事情。保证多条消息原子性地写入到目标分区，同时也能保证 Consumer 只能看到事务成功提交的消息。

设置事务型 Producer 的方法满足两个要求即可：开启 enable.idempotence = true，设置 Producer 端参数 transactional. id。最好为其设置一个有意义的名字。代码需要调整类似如下：

producer.initTransactions(); try { producer.beginTransaction(); producer.send(record1); producer.send(record2); producer.commitTransaction(); } catch (KafkaException e) { producer.abortTransaction(); }

这段代码能够保证 Record1 和 Record2 被当作一个事务统一提交到 Kafka，要么它们全部提交成功，要么全部写入失败。

实际上即使写入失败，Kafka 也会把它们写入到底层的日志中， Consumer 还是会看到这些消息。因此在 Consumer 端，读取事务型 Producer 发送的消息也是需要一些变更的。修改起来也很简单，设置 isolation.level 参数的值即可。当前这个参数有两个取值：

read_uncommitted：这是默认值，表明 Consumer 能够读取到 Kafka 写入的任何消息，不论事务型 Producer 提交事务还是终止事务，其写入的消息都可以读取。很显然，如果用了事务型 Producer，那么对应的 Consumer 就不要使用这个值。

read_committed：表明 Consumer 只会读取事务型 Producer 成功提交事务写入的消息。当然了，它也能看到非事务型 Producer 写入的所有消息。
