kafka 概览

Kafka的分类

名称 描述 特点
Apache Kafka 社区版 Kafka，Apache 基金会孵化并最终成为顶级项目 开发人数最多、版本迭代速度最快的 Kafka，劣势在于仅仅提供基础组件，对 Kafka Connect ，社区版 Kafka 只提供一种连接器。
Confluent Kafka Jay Kreps、Naha Narkhede 和饶军离开 LinkedIn 创办的Confluent 公司，专注于提供基于 Kafka 的企业级流处理解决方案 免费版包含 Schema 注册中心和 REST proxy 两大功能，以及更多连接器。收费版最有用的当属跨数据中心备份和集群监控两大功能。
Cloudera/Hortonworks Kafka Cloudera 提供的 CDH 和 Hortonworks 提供的 HDP 是大数据平台，里面集成了目前主流的大数据框架，包括集成 Apache Kafka，因此把这两款产品中的 Kafka 称为 CDH Kafka 和 HDP Kafka，2018 年 10 月两家公司宣布合并。 大数据平台天然集成了 Apache Kafka，通过便捷化的界面操作将 Kafka 的安装、运维、管理、监控全部统一在控制台中。缺点是用户对Kafka掌控性不好以及跟新后滞性较为突出。

Kafka版本号

  1. 0.7版本：只提供基础消息队列功能。
  2. 0.8版本：引入副本机制，具备真正意义高可靠分布式消息队列解决方案。
  3. 0.9版本：增加基础安全认证和权限功能，Java重新编排消费API，引入Kafka Connect。
  4. 0.10版本：引入Kafka Stream，正式升级为流处理平台。
  5. 0.11版本：提供幂等性Producer API以及事务API，对消息格式做了重构。从这个的版本开始比较稳定，功能也比较齐全，建议使用版本最低也是这个版本或者以上
0.11.x版本后新增或改进功能点 描述
内部主题分区数 严格准守遵守 offsets.topic.replication.factor 值，如果Broker数量小于这个设定值，创建主题失败。之前版本取Broker数量和这个值的最小值。刚启动1台Broker创建主题，导致后续只能有一个副本。
Rebalance分配方案 尽可能保留之前分配方案，避免过多Consumer重新链接不同Broker，造成Rebalance时间过长或者资源消耗
V2版本数据压缩 与之前V1版本相比，把公共部分提取出来压缩，压缩效率更高效。但需要注意不同版本解压缩问题的避免
创建幂等Producer 指定 Producer 幂等性的方法仅需要设置一个参数即可，即 props.put(“enable.idempotence”, ture)，或 props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG， true)。
事务支持 主要是在 read committed 隔离级别上。保证多条消息原子性地写入到目标分区，同时也能保证 Consumer 只能看到事务成功提交的消息。
Leader Epoch引入 弥补高水位机制副本同步因日志截断可能带来的数据丢失
命令工具的改变 注意不同版本Producer、Consumer、Broker配合，过低可能带来消息格式转换。
coordinator底层修改 重构控制器代码，由原来凌乱不高效的多线程，改为事件驱动的单线程处理，并提高处理优先级。
unclean Leader选举参数默认值 unclean.leader.election.enabled由原来默认值true变成flase
命令工具 具体查看官网
  1. 1.0和2.0版本：主要在Kafka Streams的改进。

Kafka线上部署方案

考虑到系统与Kafka的适配性，Linux系统表现将更加优越。

内容 描述
I/O 模型的使用 Kafka 客户端底层使用了 Java 的 selector，selector 在 Linux 上的实现机制是 epoll，而在 Windows 平台上的实现机制是 select(多路复用)。因此在这一点上将 Kafka 部署在 Linux 上是有优势的，能够获得更高效的 I/O 性能。
数据网络传输效率 Linux 平台有零拷贝机制(当数据在磁盘和网络进行传输时避免昂贵的内核态数据拷贝从而实现快速地数据传输)。在 Windows 平台上必须要等到 Java 8 的 60 更新版本才能享受到这个优势。
社区支持度 社区目前对 Windows 平台上发现的 Kafka Bug 不做任何承诺。

磁盘容量规划需要考虑：新增消息数，消息留存时间，平均消息大小，备份数量、是否启用压缩。

Kafka重要配置参数

Broker参数

这里说的Broker参数是静态参数(可随时修改不重启的属于丰台参数)，必须在 Kafka 的配置文件 server.properties 中进行设置，新增、修改还是删除必须重启 Broker 进程才能令它们生效。主题级别参数的设置需要 kafka-configs 命令来修改。至于 JVM 和操作系统级别参数设置方法比较通用化。

log.dirs：非常重要，指定了 Broker 需要使用的若干个文件目录路径。没有默认值，必须亲自指定。

log.dir：只能表示单个路径，补充上一个参数使用。

只设置log.dirs，不要设置log.dir。在生产环境中一定要为log.dirs配置多个路径，用逗号分隔的多个路径，以挂在多个物理磁盘，好处是：

  1. 提升读写性能：比起单块磁盘，多块物理磁盘同时读写数据有更高的吞吐量。
  2. 实现故障转移：Kafka 1.1 版本引入新功能。以前，只要 Broker 使用的任何一块磁盘挂掉，整个 Broker 进程都会关闭。自 1.1 开始，坏掉磁盘上的数据会自动地转移到其他正常的磁盘上，而且 Broker 还能正常工作。

zookeeper.connect：使用Zookeeper集群IP和端口号，多个逗号分隔。如果是一个集群挂在多个Kafka集群，使用类似如下配置：zk1:2181,zk2:2181,zk3:2181/kafka1和zk1:2181,zk2:2181,zk3:2181/kafka2。

listeners：告诉外部连接者要通过什么协议访问指定主机名和端口开放的Kafka 服务。它是若干个逗号分隔的三元组，每个三元组的格式为<协议名称，主机名，端口号>。

这里的协议名称可能是标准的名字，比如 PLAINTEXT 表示明文传输、SSL 表示使用 SSL 或 TLS 加密传输等；也可能是自己定义的协议名字，比如CONTROLLER: //localhost:9092。

advertised.listeners：这组监听器是 Broker 用于对外发布的，格式与上同理。

host.name/port：不要为它们指定值，过期的参数。

listener.security.protocol.map：一旦自己定义协议名称，必须指定。告诉这个协议底层使用了哪种安全协议，比如指定listener.security.protocol.map=CONTROLLER:PLAINTEXT表示CONTROLLER这个自定义协议底层使用明文不加密传输数据。

auto.create.topics.enable：是否允许自动创建 Topic。集中化管理下建议设置false

unclean.leader.election.enable：是否允许 Unclean Leader 选举，只有保存数据比较多的那些副本Partition才有资格竞选Leader，那些落后进度太多的副本没资格做这件事,建议设置成false。

auto.leader.rebalance.enable：是否允许定期进行 Leader 选举。建议设置成false, Leader的重新选举代价很高。

log.retention.{hours|minutes|ms}：控制一条消息数据被保存多长时间。从优先级上来说 ms 设置最高、minutes 次之、hours 最低。

log.retention.bytes：指定 Broker 为消息保存的总磁盘容量大小。-1标识不做任何拦截。多租户公用物理磁盘可以使用。

message.max.bytes：控制 Broker 能够接收的最大消息大小。默认 1000012 有点少，需要自己更改。

Topic级别参数

Topic 级别参数会覆盖全局 Broker 参数的值。

retention.ms：规定该 Topic 消息被保存的时长。默认是 7 天。

retention.bytes：规定要为该 Topic 预留多大的磁盘空间。当前默认值是 -1，表示可以无限使用磁盘空间。

Topic级别参数设置有两种方式，一种是创建Topic时设置，一种是修改Topic时设置：

  1. bin/kafka-topics.sh --bootstrap-server localhost:9092 --create --topic transaction --partitions 1 --replication-factor 1 --config retention.ms=15552000000 --config max.message.bytes=5242880
  2. bin/kafka-configs.sh --zookeeper localhost:2181 --entity-type topics --entity-name transaction --alter --add-config max.message.bytes=10485760

JVM参数设置

Kafka 自 2.0.0 版本开始，正式摒弃对 Java 7 的支持了，至少使用 Java 8 。

垃圾回收器： Broker 所在机器 CPU 资源非常充裕，建议使用 CMS 收集器。启用方法是指定-XX:+UseCurrentMarkSweepGC。

否则，使用吞吐量收集器。开启方法是指定-XX:+UseParallelGC。

如果已经使用 Java 8 ，用默认的 G1 收集器。

指定堆大小：改变环境变量，KAFKA_HEAP_OPTS。

指定 GC 参数：改变环境变量，KAFKA_JVM_PERFORMANCE_OPTS

$> export KAFKA_HEAP_OPTS=--Xms6g --Xmx6g $> export KAFKA_JVM_PERFORMANCE_OPTS= -server -XX:+UseG1GC -XX:MaxGCPauseMillis=20 -XX:InitiatingHeapOccupancyPercent=35 -XX:+ExplicitGCInvokesConcurrent -Djava.awt.headless=true $> bin/kafka-server-start.sh config/server.properties

操作系统参数

文件描述符限制：通常情况下设置成一个超大的值是合理的做法，比如ulimit -n 1000000。不设置话后可能出现很多“Too many open files”的错误。

文件系统类型：据官网测试报告，XFS 的性能要强于 ext4。最近有 Kafka 使用 ZFS 的数据报告，貌似性能更加强劲。

Swappiness：可以设置成一个较小的值，不推荐网上说的设置成0。当物理内存耗尽，操作系统会触发 OOM killer 组件，会随机挑选一个进程然后 kill 掉，将无法给用户任何预警。

提交时间：或者说是 Flush 落盘时间。默认是 5 秒才会从提交到真正落盘，自己根据提交性能和可能的数据丢失(有多副本保障)综合决定。
